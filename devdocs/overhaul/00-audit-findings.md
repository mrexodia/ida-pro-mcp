## api_analysis.py  -  Read-only code-analysis surface of the MCP server: Hex-Rays decompilation, annotated disassembly, function profiling, deep per-function batch analysis, cross-reference queries (address + struct-field), single-level callees, bounded call graph, byte/immediate/string/ref search, semantic instruction search, basic-block CFG extraction, and function export. This is the module that underpins the proposed nested "russian-doll" call view (axis 2): callees/callgraph/basic_blocks/xref_query/decompile are its building blocks.
### Weaknesses / bugs:
 - decompile() decompiles the function TWICE: once via decompile_function_safe (line 776) for text, then again via ida_hexrays.decompile(start) (line 784) purely to collect refs. The cfunc from the first pass is discarded. This doubles the cost of the single most-used tool and there is zero caching across calls.
 - No decompiler cache invalidation / refresh control anywhere. After rename/set_type/declare_type the Hex-Rays cache can be stale; there is no force-refresh or mark_cfunc_dirty exposure here, and force_recompile lives elsewhere. Callers get stale pseudocode with no signal.
 - get_callees() in utils.py (used by analyze_batch line 1212) uses idc.find_func_end(func_start) instead of func.end_ea, so for tail-chunked / non-contiguous functions it walks only the first chunk and silently misses callees in other chunks. The in-module callees() tool uses func.end_ea and thus disagrees with analyze_batch's callee list — inconsistent results for the same function.
 - callees() and _collect_callees_for_function only detect direct/static call targets and only via call mnemonics or CodeRefsFrom; tail-call jmps (jmp into another function) and indirect/vtable calls are missed. The docstring admits indirect misses but tail-call jmps are not mentioned and are common in optimized code.
 - find type='immediate' paging is broken/quadratic: on every call it rescans from each executable segment's start and uses offset to skip already-seen matches (1979-2001). cursor.next=offset+limit (2013) so resuming re-decodes the entire prefix every page; for a hot constant this is O(n*pages). next_start-style resumption (as in insn_query) is not used.
 - find type='immediate' break logic only honors 'more' to stop, but seen_insn dedup is per-call only; combined with offset skipping, a match can be counted toward skipped in one call and re-counted in the next because seen_insn is rebuilt each call — offset semantics are unstable across pages.
 - xrefs_to_field calls ida_typeinf.get_udm_by_fullname(None, ...) (line 1516) passing None for the til even though a valid til was fetched at 1485; relying on the default library can fail for types in a local til and is inconsistent with the get_named_type(til, ...) call just above.
 - _collect_line_comments concatenates E_PREV/regular/repeatable/E_NEXT comments into a flat list with no labels (508-530); the consumer cannot tell an anterior comment from a posterior one, and a regular cmt equal to the repeatable is deduped but anterior==posterior duplicates are not.
 - disasm() with include_total=true still stops walking once total_count is counted but the per-line work short-circuits; however 'more' is recomputed at 906-907 as total_count>offset+max_instructions which double-counts: _maybe_add already set more=True in the page-full path, and the recompute can flip more back inconsistently if the function ended exactly at the page boundary.
 - export_funcs JSON path appends partial func_data on a mid-record exception only via the outer try; but c_header/prototypes post-processing iterates results and reads func['prototype'] — error entries lack that key so they are silently dropped, matching the pitfall note, but name-bearing functions with empty prototype are also dropped with no diagnostic count.
 - find_bytes/find string mode do ea += 1 after each match then call searcher again from ea — correct for non-overlapping but the more-check at limit (1730) calls searcher(ea+1) a second time, doing an extra full scan step just to set a boolean; cheap but the same pattern is duplicated in 3 places.
 - callgraph traverse() is recursive (2527-2577) with no explicit recursion-depth guard beyond max_depth; deep linear callee chains near max_depth=very-large combined with Python's recursion limit can raise RecursionError that is caught only by the outer per-root try, losing the whole graph instead of truncating gracefully.
### Missing capabilities:
 - Decompiler refresh / cache invalidation control (mark_cfunc_dirty, hexrays clear_cached_cfuncs) so pseudocode reflects recent rename/retype edits.
 - Bidirectional line<->address mapping as a first-class result: decompile only appends /*0xNNNN*/ markers in text; there is no structured map of pseudocode line -> [eas] and ea -> line for programmatic pivoting (key for axis 2 finer view).
 - Per-statement / per-expression ctree access (microcode, citem types, local variable list, lvar types) — only flat text + object refs are exposed. No way to enumerate decompiled local variables or their types.
 - Microcode / mba access for advanced analysis (ida_hexrays.gen_microcode), and decompiler warnings (hexrays_failure_t / cfunc warnings) are not surfaced.
 - Callers-side recursion is absent from this module (only callgraph descends callees; callers_recursive lives in api_graph). A combined In/Out russian-doll view (axis 2) is not assembled here.
 - Switch/jump-table and indirect-call resolution (idaapi.get_switch_info / processor xref hints) for callees and callgraph edges.
 - No reachability/path tool in this module (reaches/path lives in api_graph/api_composite); the module's callgraph cannot answer 'does A reach B' or return paths.
 - String/immediate search cannot scope to a function or segment (find always scans whole image / all exec segs); insn_query can scope but find cannot, an inconsistency.
 - No regex/text disasm search here (find_regex/search_text are elsewhere) despite disasm producing the text — a scoped 'grep the disassembly' is missing.
 - find_bytes cannot scope to a range/segment/function (always min_ea..max_ea) and offers no alignment / executable-only filter.
 - No demangled-name option on xref fn resolution; get_function returns raw names only.
 - basic_blocks does not expose per-block instruction list, terminating-branch mnemonic, or loop/back-edge classification — the caller must re-decode to find loop bodies despite the PRO-TIP suggesting loop detection.

### Naming/doc issues:
 - find: tool name is extremely generic for a 4-mode search (string/immediate/data_ref/code_ref). Modes overlap with xref_query (data_ref/code_ref) and find_bytes/insn_query (immediate). Either split into find_string/find_immediate or rename to find_value; the single 'find' is ambiguous in a 100+ tool surface (axis 4).
 - callees vs callgraph vs xrefs_to naming does not convey directionality clearly: 'callees' is outbound-direct, 'callgraph' is outbound-recursive, but the reverse direction is spread across xrefs_to/xref_query/callers_recursive in other modules — no symmetric callers() here. Inconsistent mental model for the russian-doll view.
 - decompile docstring claims it is THE primary tool but never mentions the refs list can be incomplete (only cot_obj objects are visited, not all globals) nor that results may be stale after edits.
 - disasm docstring says refs make it 'strictly more informative than plain text' but does not document that include_total forces a full second walk cost in the same call (it is in PITFALL but the cost interaction with offset is understated).
 - func_profile 'size' is hex string while internal size_int is dropped (1054); docstring/RETURNS does not state size is a hex string, easy to misparse.
 - xrefs_to message 'No cross-references to this address' is reused verbatim in xref_query even for direction='from' (line 1445) where it should say 'No cross-references from/to'.
 - insn_query op0/op1/op2/op_any are documented as 'int operand value' but for o_mem/o_near operands the matched value is op.addr and for o_imm it is op.value — the docstring doesn't tell the user which operands compare against addresses vs immediates, making operand matching guesswork.
 - find_bytes pattern doc says '??' wildcard but does not state byte-level only (no nibble wildcards) nor case requirements; relies on compat.make_bytes_searcher behavior that is undocumented at the tool level.
 - basic_blocks 'type' field is the raw IDA fc_block_type_t integer with no enum decoding in output or docstring — callers get an opaque int.

## api_core.py  -  Core IDB metadata + basic queries: server health/warmup, function listing/lookup/query, globals, imports, unified entity query, integer conversion, IDB save, string regex search, rendered-listing text search, and the session strings cache (init_caches).
### Weaknesses / bugs:
 - int_convert auto-size bug: signed to_bytes with bitlength-derived size raises spurious OverflowError for values that need the sign bit (e.g. 128/0x80), and bare-string default 'size=64' is consumed as 64 BYTES not 64 bits, contradicting the param doc (lines 536, 552-570).
 - Pervasive lack of caching for functions/imports: list_funcs, func_query, list_globals, entity_query, imports, imports_query each re-enumerate the whole image (and entity_query 'names' re-walks imports twice) — O(N) per call, no equivalent of _strings_cache for functions/imports (axis 8).
 - Silent empty-result swallowing of errors: invalid name_regex in func_query, invalid regex propagating as raw exception in find_regex, bad min_addr/max_addr in entity_query — inconsistent error contracts vs search_text's structured error (axes 4/8).
 - No total count on list_funcs/func_query/list_globals/imports/imports_query Pages (only entity_query has it) — inconsistent pagination surface; bulk '*' lookup_funcs truncates at 1000 with no truncated flag.
 - list_globals mislabels code labels (loc_*) as globals and omits size/segment/type that the parallel _collect_entities('globals') already computes (axes 4/8).
 - idb_save is_idaq() exception fallback biases toward the DESTRUCTIVE headless DBFL_KILL branch (line 1012-1013) and save-as can overwrite an existing unrelated file without consent (axis 7).
### Missing capabilities:
 - No segments/sections listing tool here (start/end/perm/class/bitness) despite _exec_segments/_all_segments helpers existing — segment enumeration is only reachable indirectly (axis 2/8). (api_core's stated scope includes segments and entry points but exposes neither.)
 - No entry-points tool (idautils.Entries / ida_entry) even though the prompt lists entry points as in-scope for this module.
 - No demangled-name search / mangled<->demangled resolution in lookup_funcs (ida_name demangling).
 - No function-prototype / signature in function rows (calling convention, arg count) — has_type is a bare boolean; cannot distinguish guessed vs user types.
 - No string TYPE/encoding info (C/Pascal/UTF-16) or string length-class in find_regex/strings rows; idautils.Strings carries strtype but it is dropped (line 152).
 - No imagebase-relative (RVA) addressing option anywhere; all addrs are absolute hex.
 - No metadata/idb-info tool (processor, compiler, bitness, file format, ABI) — only partially surfaced in server_health.
 - No cache-invalidation tool or auto-invalidation after rename/define so list_globals/strings can go stale within a session (invalidate_strings_cache exists but no global wiring is visible from here).

## api_types.py  -  Struct/enum creation, type inference, type application (global/local/stack/function), local type library inspection and querying.
### Weaknesses / bugs:
 - Likely real bug: apply_tinfo (set_type L994/L1020) is called with PT_SIL, a PARSE flag, where a TINFO_* apply flag (TINFO_DEFINITE) is expected - applied types may be non-definitive/weak.

 - No dry-run/preview for the DESTRUCTIVE type-apply tools, and type_apply_batch is explicitly non-transactional with no rollback (axis 7/8).
 - Redundant get_udt_details calls (search_structs L542-554) and redundant udm fetches (set_type L1081-1084) waste work.

 - Type creation from a memory layout / 'create struct from selection' (struct synthesis from observed access patterns) - a common 9.x RE workflow.
 - Importing standard type libraries (til) / add_til, and listing loaded type libraries.
 - Applying a type to a RANGE / array of N elements at an address (array stamping).

## api_memory.py + api_stack.py  -  api_memory exposes batch static reads (raw bytes, typed ints, string literals, type-aware globals) and two binary-mutating writers (patch raw hex, put_int typed). api_stack exposes stack-frame layout reads plus declare/delete of named typed stack variables.

## api_probes.py (PROBE / WATCH / AUTOPILOT toolkit)  -  The axis-2 powerhouse: a non-stopping live-debugger instrumentation layer. Installs Python-condition breakpoints that capture register/stack/memory specs into a ring buffer and ALWAYS return False (never halt the debuggee), plus watchpoints, call tracing, import probes, live struct reads, appcall, live memory scan, best-effort snapshots, and a whitelisted autopilot sequencer. Pure-logic helpers (parse_capture_spec, parse_ptr_chain, plan_autopilot) are split out so they unit-test headless without idaapi.
### Weaknesses / bugs:
 - eval() injection dressed as a sandbox (line 789, _eval_predicate): `eval(predicate, {'__builtins__': {}}, env)` runs user-supplied condition/predicate strings. The `# noqa: S307 (sandboxed env)` comment is FALSE — emptying __builtins__ does not sandbox Python; `(c).__class__.__mro__[1].__subclasses__()` style escapes reach os/subprocess. The predicate originates from probe_add(condition=...) / watch_*(predicate=...). On a local single-user IDA the trust boundary is the MCP client, but the code claims safety it does not provide. Either use ast.literal_eval-based restricted evaluation, a real AST whitelist, or a simple comparison mini-language.
 - trace_calls capture_ret and probe_net pre_post are FAKE pairings (lines 1419-1428, 1855-1860): they advertise return-value / before-after capture but only append an English instruction string to the capture list. No return-site probe is installed and no return value is ever recorded. The docstrings ('paired entry+return probes') are materially inaccurate (axis 2 + axis 4).
 - run_until status corruption (lines 1139-1142): presence of ANY drained record forces status='hit', clobbering the true 'exited'/'timeout' outcome — callers lose the resume result.
 - snapshot_restore reports restored:True on partial/total failure (lines 2105-2117) because every write error is swallowed; and it has no same-process identity check, so restoring into a relaunched process writes bytes/regs to wrong addresses, potentially crashing the debuggee.
 - watch_field/watch_region silently miss writes: the single-HW-slot model (watch_region, 1303/1276-1280) means writes outside the armed aligned slot never fire; combined with NO detection of DR-slot exhaustion (only 4 slots), a 5th watch can fail silently. Change-detection also relies on the bpt firing at all — if the slot doesn't cover the byte, the change-detector never runs.
 - _eval_capture stack args assume 32-bit cdecl at callee entry (lines 567-582): a 64-bit target passes the first 4 args in rcx/rdx/r8/r9 (Win64) — argN read off [rsp+8*(n+1)] is WRONG on x64. The toolkit detects ptr size (8) but still reads args from the stack, so x64 register-passed args are garbage. argN is only correct on x86.
 - Triple spec write + lock churn in watch_field/watch_region (e.g. 1209-1248): spec is written, overwritten by the installer, then re-merged across three lock acquisitions; watch_last is seeded twice; easy to desync under concurrency.
 - Probe records are never garbage-collected per-probe: probe_clear (1045-1075) removes the bpt+registry+spec but leaves the captured ring records, so probe_drain/trace_summary still surface records for deleted probe_ids.
 - No timeout/backpressure on a hot probe between run_until calls: _probe_dispatch records every hit with no every_nth honoring (despite the param) and no rate limit; a tight-loop bpt can saturate the ring and inflate 'dropped' with no automatic self-disarm beyond max_hits.

## api_python.py  -  Axis-6 Python injection bridge. ## ida-domain-sdk  -  Capability gap analysis mapping IDA 9.3 / ida-domain SDK features against the current MCP server's exposed tools, prioritising the owner's emphasis on debugging and on understanding linked structures (call hierarchies, memory, pseudocode/ctree, microcode). Derived from reading the ida-domain llms-full.txt reference (Functions, LocalVariables, Pseudocode/ctree, Microcode, Xrefs, Hooks: Debugger/Decompiler, Flowchart, Bytes, Types) cross-referenced with the project's api_debug/api_probes/api_analysis/api_graph/api_modify/api_domain modules.
### SDK capabilities (gap analysis):
 - [hexrays|exposed=no|val=high] ida-domain PseudocodeFunction exposes the full decompiler ctree with typed node queries: find_calls(target), find_loops, find_if_instructions, find_assignments, find_objects, find_strings, find_variables(by index/name), walk_expressions/walk_instructions, and find_parent_of. This turns pseudocode into a queryable AST instead of opaque text. The project currently only returns decompiled TEXT (api_analysis.decompile) plus regex/heuristic line scraping; it does not walk the ctree. => pseudocode_query(func, kind in {calls,loops,assignments,ifs,variable_uses,strings,objects}, target=...) returning structured nodes with ea, line, parent, and the rendered subexpression — the backbone of true 'pseudocode comprehension' (axis 2).
 - [hexrays|exposed=no|val=high] Functions.get_local_variables + get_local_variable_references returns each lvar's references annotated with LocalVariableAccessType (READ/WRITE/ADDRESS) and LocalVariableContext (ASSIGNMENT, CALL_ARG, COMPARISON, CONDITION, POINTER_DEREF, ARRAY_INDEX, RETURN, ...), each with ea/line/code_line. This is precise def-use information at decompiler level. The project's trace_data_flow is heuristic/disasm-based and there is no lvar use-classification. => lvar_usage(func, var) -> per-reference {access:read|write|addr, context, ea, line, code} plus a summary (where written, where read, where passed as arg) — directly serves memory/variable comprehension (axis 2).
 - [hexrays|exposed=no|val=medium] db.microcode.generate / get_text / from_decompilation / generate_for_range expose Hex-Rays microcode at selectable maturity (GENERATED..LVARS), with a full object model (MicroBlock, MicroInstruction, MicroOpcode, MicroCallInfo, MicroLocalVars, MicroGraph). Microcode is the cleanest IR for value-tracking, call-arg resolution, and de-obfuscation. The server exposes nothing at the microcode layer. => microcode_text(func, maturity) for a readable IR dump, and microcode_calls(func) to enumerate resolved call args/return types per call site — feeds both pseudocode comprehension and value-flow (axis 2/8).
 - [xrefs|exposed=partial|val=high] The pieces exist separately: callgraph/callees_recursive/callers_recursive (api_graph) for breadth, basic_blocks for branch structure, and the ctree for in/out calls + guarding conditions. ## design:debug-wishlist  -  Design the DEBUG-RE feature set: assess the current api_debug.py + api_probes.py + trace.py surface against the owner's 8 priority RE capabilities (snapshots, probes/sondes, watches/espions, conditional breakpoints, traces, memory comprehension, pseudocode comprehension, russian-doll module/large-function call-hierarchy) and specify concrete new MCP tools (names/args/return shape) to close the gaps.
### Weaknesses / bugs:
 - trace_calls(capture_ret=True) does NOT actually capture the return value - it only annotates the ProbeRef with a string telling the human to manually place a probe_add(['ret']) at the return site (api_probes.py:1419). Documented as a 'known limitation' but it means the headline call-tracing feature is half-built.
 - every_nth is accepted by probe_add but is purely advisory - the dispatcher records EVERY hit (api_probes.py:980, doc confirms). There is no real sampling, so hot functions can only be throttled via max_hits or a condition, not by a 1-in-N sampler.
 - snapshot_save/restore are in-process dicts only (api_probes.py:2014) - lost when the MCP server restarts, not stored in the IDB, no diffing, no auto-capture of touched ranges. No use of IDA's real ida_dbg take_memory_snapshot / session snapshots.
 - No conditional-breakpoint HIT COUNTER: dbg_set_bp_condition supports an IDC/Python expr but there is no native 'break on Nth hit' / 'count hits without breaking' - you must hand-write a counter into the condition.
 - _eval_predicate / breakpoint Python conditions eval() user strings; predicate is sandboxed ({'__builtins__':{}}) but probe condition strings written onto the bpt run in IDA __main__ with full builtins. Low risk (operator-supplied) but worth an explicit note.
 - No register WATCH (only memory watch_field). 'watch EAX changing' or 'flag ZF' has no tool - you'd need a code probe at every write site.
 - No exception/signal catch configuration (ida_dbg.set_exception_info / EXC handling) - cannot tell the debugger to catch/pass first-chance exceptions, a core RE need for packed/anti-debug targets.
 - _stack_arg assumes a clean cdecl/thiscall stack frame at entry; for x64 register-arg ABIs argN reads the stack home slots which are often uninitialized at entry. Documented, but there is no x64 fastcall (rcx/rdx/r8/r9) arg helper at all.
 - Module memory-map comprehension is read-only inside memory_scan (_live_memory_ranges, api_probes.py:1874) but never exposed as a tool - an agent cannot list the live memory map, module bases, or page protections.
 - No single 'context at stop' tool: after a hit an agent must call dbg_gpregs + dbg_stacktrace + dbg_read + disasm separately. No fused register+stack+disasm+pseudocode-line view.
### Missing capabilities:
 - Live memory map / module list / page protections as a tool (data exists internally in _live_memory_ranges but is unexposed).
 - Return-value capture in call tracing (only a manual-TODO annotation today).
 - Nested call-hierarchy / call-tree reconstruction with depth (the owner's russian-doll ask - no tool).
 - Dynamic branch/condition coverage (which Jcc edges a function actually takes at runtime).
 - Register and CPU-flag watches (only memory watchpoints exist).
 - Counting / Nth-hit breakpoints and count-only execution-frequency probes.
 - Snapshot diffing, auto-range snapshots, and IDB-persistent snapshots.
 - Exception/signal first-chance catch configuration (ida_dbg exception info).
 - Live pseudocode overlay with register/lvar values plugged in (Hex-Rays line<->ea + lvar locations).
 - Fused stop-context tool (regs+flags+stack+disasm+pointer classification in one call).
 - Real x64 register-ABI argument resolution (argN is 32-bit-stack-only).
 - True statistical sampling (every_nth is a no-op).
 - Ordered run timeline / record-replay approximation (only grouped rollups exist).
 - Pointer-deref capture tokens for one-hop object comprehension.

## design:russian-dolls  -  Design a nested "russian-doll" call-hierarchy comprehension feature (axis 2): given a module / large function / class, produce both a WIDER view (transitive In/Out calls across N levels) and a FINER view (intra-function control flow: branches, jump conditions, which calls fire on which path), letting an analyst zoom from a subsystem down to a single guarded call and back up, with optional links to decompiler pseudocode and live-debug edge annotation.
### Weaknesses / bugs:
 - No bidirectional rooted view: to see both In and Out around one function an analyst must call callers_recursive AND callees_recursive separately and stitch results by hand; depths are then in two disjoint coordinate systems (both 0 at root).
 - callgraph (api_analysis) and callees_recursive (api_graph) both compute the downward closure but with different edge semantics (callgraph follows ALL CodeRefsFrom into any function including jumps/tail-calls; _direct_callers filters to NN_call* but _direct_callees does NOT) — inconsistent edge definitions across the two 'same' directions, which will bite a nested view that mixes them.
 - _direct_callees follows CodeRefsFrom and counts any code ref landing in a function as a callee, so tail-call jumps and fallthrough-into-next-function can masquerade as calls; the finer view needs to distinguish call vs jump vs fallthrough explicitly.
 - No intra-function guard model anywhere: which call is gated by which branch condition is completely absent. decompile gives text; basic_blocks gives a CFG with no condition labels. This is the core gap for the FINER view.
 - Indirect/virtual calls are invisible to every static tool here (documented as a PITFALL repeatedly) — a russian-doll view of a vtable-heavy class will be misleadingly sparse unless it surfaces indirect call sites as explicit 'unresolved' leaf nodes.
 - callgraph.traverse records the node then checks depth>max_depth on entry, and recurses inside the edge loop (DFS not BFS despite 'breadth-limited' docstring) — depth labelling can differ from api_graph's true BFS; a unified tool should standardise on the BFS helper.
### Missing capabilities:
 - Bidirectional (In+Out) rooted call view in a single result with a shared depth/coordinate system (only separate callers_recursive / callees_recursive exist).
 - Intra-function control-flow model that labels branch conditions and maps which call fires on which path / under which guard (no tool exposes guarded calls).
 - Strongly-connected-component / recursion-cluster collapse for the wide view.
 - Honest per-edge kind classification (call vs tailcall vs jump vs fallthrough vs indirect) shared across the graph tools.
 - Surfacing indirect/virtual call sites as explicit unresolved nodes so vtable-heavy classes are not silently sparse.
 - Decompiler-ctree-derived guard expressions and pseudocode line anchors attached to CFG blocks.
 - Runtime overlay merging actually-taken edges/branches onto the static hierarchy.
 - A drillable cursor/handle protocol that lets a bounded outer view be expanded at finer granularity in-place.

## tests  -  Custom in-IDA test framework (framework.py) plus ~24 test files exercising the MCP tool surface against two ELF fixtures (crackme03.elf general regression, typed_fixture.elf typed-globals/structs/stack). Tests register via @test(binary=...) and run headless through `ida-mcp-test`. Includes a rich assertion library (assert_shape/assert_typed_dict/assert_ok/assert_error/one_of/list_of/optional), live-IDB round-trip tests (trace netnode, comments), and mock-based unit tests for the debugger control flow that cannot rely on a live process.
### Weaknesses / bugs:
 - api_probes.py has ZERO test coverage despite containing substantial deterministic, debuggee-free helper logic: parse_capture_spec, parse_ptr_chain, build_probe_record, _stable_probe_id, validate_autopilot_step, plan_autopilot. These are pure parsers/planners that can and should be unit-tested without any running process. This is the single biggest gap given axis-2 is the headline feature area.
 - No test asserts that probe/watch/run_until/trace_calls/probe_api_call fail cleanly when NO debugger is attached. _require_debugger() is the universal guard for the whole live-RE toolkit and is completely unverified - a regression there would silently break every dbg tool with a confusing error.
 - Several whole modules have no test file at all: api_graph.py, api_recipes.py, api_domain.py, api_docs.py, api_survey.py (only test_api_survey exists - need to confirm depth), api_composite partially. The in-tool docs corpus (27 files) and search_docs are unverified for link integrity.
 - The @ext('dbg') visibility gating (tools hidden unless ?ext=dbg) is not asserted by any test in the debug/probe path - test_discovery/test_http reference 'probe' only for instance-discovery TCP probing, not tool-extension gating. A bug exposing dbg tools to non-dbg clients would pass CI.
 - Mock-based debug tests are tightly coupled to current internal symbol names (_DbgStartBatchHook, _arm_dbg_start_batch_hook, _DBG_START_IP_GRACE_POLL_COUNT, get_pre_call_batch). They verify wiring faithfully but will not catch behavioral drift in real IDA dbg APIs, and are brittle to refactors - acceptable but worth noting.
 - No assertion anywhere that mutating/patch tools (patch, patch_asm, put_int, make_data) refuse to touch the analysed binary without explicit consent (axis-7). The test suite does not encode the 'never patch unless asked' invariant, so a guardrail regression is invisible to CI.
 - trace round-trip tests reuse 'server_health' as the traced tool; they exercise the trace backend well but never trace an actual IDA-mutating tool to confirm argument capture of complex/binary payloads beyond the single py_eval case.
### Missing capabilities:
 - No live-debuggee integration harness: IDA 9.x can run a local debugger on the fixture ELF headlessly, but the suite never launches a process, so the entire snapshot/probe/watch/trace_calls/appcall runtime path is unexecuted end-to-end.
